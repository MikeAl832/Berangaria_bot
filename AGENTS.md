# AGENTS.md

This file applies to the entire repository.

## Project overview

Berangaria is a Python 3.11 Telegram bot built on `python-telegram-bot`. DeepSeek handles chat and summarization, Gemini handles vision/audio and embeddings, Mem0 provides long-term memory, Qdrant stores vectors, and SQLite persists conversation history and runtime settings.

Key modules:

- `main.py`: application startup, handler registration, scheduled jobs, and graceful shutdown.
- `handlers.py`: Telegram commands, message buffering, media handling, access control, and chat-event handling.
- `llm_client.py`: history rendering, memory retrieval/flush, summarization, LLM retries, tool rounds, reply delivery, and persistence.
- `streaming.py`: DeepSeek SSE reconstruction and throttled Telegram draft/status previews.
- `state.py`: shared in-memory state, per-chat locks, and SQLite persistence.
- `tool_handlers.py` / `tools.py`: LLM tool dispatch, web search, URL reading, reactions, replies, and stickers.
- `vision_provider.py`: Gemini image, video, and audio requests.
- `memory_store.py`: retryable Mem0 initialization.
- `sticker_store.py`: Gemini embeddings and Qdrant-backed sticker search/indexing.

## Setup and validation

Use Python 3.11, matching CI and Docker:

```bash
uv venv .venv --python 3.11
uv pip install --python .venv/bin/python -r requirements.txt pytest ruff bandit
```

Tests supply placeholder API keys in `tests/conftest.py`; never add real credentials to tests or the repository.

Run before handing work off:

```bash
.venv/bin/python -m py_compile $(git ls-files '*.py')
.venv/bin/pytest -q
.venv/bin/ruff check . --exclude .venv
.venv/bin/bandit -q -ll -r . -x ./.venv,./tests
git diff --check
```

Bandit may report intentional low-severity best-effort exception handling and non-cryptographic randomness. Any medium/high finding must be resolved.

## Concurrency and history invariants

- A complete chat turn is serialized with `state.get_turn_lock(history_key)`: append user input, call LLM/tools, deliver the reply, then persist the assistant entry.
- Whenever both locks are needed, acquire the turn lock before `state.get_history_lock`. Never reverse this order.
- Commands or jobs that clear, summarize, replace, or delete history must use the same turn lock. Reactions also use it before mutating assistant entries.
- Do not hold the narrow history lock across network I/O. The turn lock protects the logical transaction; the history lock protects short in-memory/SQLite mutations.
- Group debounce buffers are per user, but group history is shared. Do not remove per-history turn serialization.
- Summarization must work on copies of retained entries. Never renumber or otherwise mutate live history before the API request succeeds.
- Deliver Telegram text before adding the assistant entry to history. A delivery failure must not create a response the user never received or retry the whole LLM turn.

## Async and external-I/O rules

- Never call blocking HTTP, Qdrant, embedding, or search code directly from an async handler. Use an async client or `asyncio.to_thread`.
- Apply `_check_access_permissions` before downloading media or calling Gemini. This must be true for every media type, including photos and stickers.
- `tools.read_url` handles untrusted model/user URLs. Keep HTTP(S)-only validation, reject non-global IPs, disable automatic redirects, and revalidate every redirect target.
- Keep transport/API retries separate from LLM tool-call round limits. Tool use must not consume network recovery attempts, and every loop must have an explicit terminal error.
- Tool arguments are untrusted JSON generated by a model. Parse defensively and return a tool error rather than replaying already executed actions.
- Streaming previews may expose only final-answer `delta.content`. Never send `reasoning_content`, tool names, tool arguments, or raw SSE events to Telegram.
- Private chats use `send_message_draft`; direct group turns may reuse a throttled preview message. Ambient group turns must not create a preview because they may legitimately end in silence.

## Persistence and memory rules

- SQLite writes are write-through for history changes. A reported reset or clear must be durable before confirming it to the user.
- Use `memory_store.memory` dynamically; do not import the object by value. Startup retries initialization because Qdrant may not yet be ready in Docker.
- Failed Mem0 batches must be retried and returned to `state.pending_memory`; never clear a batch permanently before successful storage.
- Do not commit `.env`, API keys, bot tokens, `bot_data/`, Qdrant storage, databases, logs, or generated virtual environments.

## Configuration and deployment

- `config.yaml` is the runtime source of non-secret defaults. Environment variables override secrets and selected deployment paths.
- `admin_alert_chat_id` is one integer Telegram chat ID or `null`, not a list.
- Docker uses `BOT_DB_PATH=/data/bot_state.db` and `BOT_LOG_FILE=/data/bot.log`; preserve the mounted `/data` volume for stateful changes.
- Keep dependency and container-version changes deliberate. Mention migrations or Qdrant embedding-dimension changes prominently in the handoff.

## Change discipline

- Preserve unrelated user changes in a dirty worktree.
- Add regression tests for bug fixes, especially races, failure ordering, security boundaries, and persistence behavior.
- Prefer focused patches over broad rewrites of `handlers.py` or `llm_client.py` unless the behavior is protected by tests.
- Update `README.md` or `CONFIG_README.md` when configuration, startup, commands, deployment, or externally visible behavior changes.

# Строгий fail-closed пайплайн долговременной памяти

Статус: принято.

## Контекст

Передача сырого диалога в Mem0 позволяла extractor'у сохранить вывод, шум или неверную атрибуцию до последующей модерации. Fail-open поведение при сбое модератора прямо противоречит требованию хранить только подтверждённые устойчивые самоутверждения.

## Решение

Каждое непустое исходное текстовое сообщение становится отдельным элементом SQLite-очереди с областью чата, автором, Telegram message ID и временем — короткие и low-signal реплики тоже проходят окончательное решение verifier'а. Более поздняя правка Telegram не заменяет уже зафиксированный источник; пересылки и сгенерированные описания медиа в очередь не входят. TikTok-ссылки сохраняются как часть оригинального provenance, но удаляются из копии текста до любого LLM-вызова.

DeepSeek создаёт структурированные кандидаты, после чего отдельный verifier проверяет самоатрибуцию, устойчивость и точный доказательный фрагмент. Verifier получает актуальные факты того же субъекта и обязан повторно использовать существующий `fact_key` для того же свойства. Детерминированные проверки требуют буквальную подстроку без нормализации пробелов и отклоняют malformed output, модальность и чувствительные категории, включая структурные признаки точного адреса. Любое сомнение или техническая ошибка закрывает путь к записи. Решение и краткая причина пишутся только в технический журнал.

Сначала verifier принимает окончательное решение по всем кандидатам источника; только затем одобренные факты передаются в Mem0 по одному с `infer=False`. Все Mem0 writes одного источника публикуются в SQLite одной транзакцией. Частичный сбой удаляет новые vectors и восстанавливает прежние версии до retry. Mem0 служит только векторным индексом точного текста. Комбинация `(scope, subject_id, fact_key)` имеет одну актуальную версию; новое ясное самоутверждение обновляет прежний vector ID in-place. Поиск дополнительно сверяет ID и точный текст результата с SQLite-реестром одобренных фактов той же области чата.

До завершения соответствующего буферизованного хода источник имеет статус `waiting`, блокирует более новые элементы FIFO и недоступен worker. После доставки обычного ответа фиксированный снимок ID становится `pending`, поэтому память не задерживает Telegram reply и отменённый debounce-task не освобождает более новые сообщения. Если процесс перезапущен до доставки, `waiting`-источник становится `abandoned`, не проверяется и теряет сырой текст. Временная ошибка блокирует более новые источники, чтобы запоздалый retry не перезаписал более свежий факт. После ровно пяти технических сбоев источник становится dead-letter. Полный сырой текст очищается после окончательного решения; для одобренного факта остаются только точная цитата и provenance.

## Последствия

Точность и воспроизводимость важнее полноты: неоднозначные факты будут потеряны, а новые воспоминания появятся с задержкой и потребуют двух LLM-вызовов. Зато сбой extractor'а, verifier'а, Mem0 или Qdrant не создаёт память, чатовые области не смешиваются, а Mem0 не может самостоятельно расширить одобренную формулировку.

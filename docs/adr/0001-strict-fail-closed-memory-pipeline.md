# Строгий fail-closed пайплайн долговременной памяти

Статус: принято.

## Контекст

Передача сырого диалога в Mem0 позволяла extractor'у сохранить вывод, шум или неверную атрибуцию до последующей модерации. Fail-open поведение при сбое модератора прямо противоречит требованию хранить только подтверждённые устойчивые самоутверждения.

## Решение

Каждое непустое исходное текстовое сообщение становится отдельным элементом SQLite-очереди с областью чата, автором, Telegram message ID и временем — короткие и low-signal реплики тоже проходят окончательное решение verifier'а. Более поздняя правка Telegram не заменяет уже зафиксированный источник; пересылки и сгенерированные описания медиа в очередь не входят. TikTok-ссылки сохраняются как часть оригинального provenance, но удаляются из копии текста до любого LLM-вызова.

DeepSeek создаёт структурированные кандидаты, после чего отдельный verifier проверяет самоатрибуцию, устойчивость и точный доказательный фрагмент. Verifier получает актуальные факты того же субъекта и обязан повторно использовать существующий `fact_key` для того же свойства. Детерминированные проверки требуют буквальную подстроку без нормализации пробелов и отклоняют malformed output, модальность, желания и чувствительные категории, включая адресные, медицинские и финансовые маркеры. Любое сомнение или техническая ошибка закрывает путь к записи. Решение и краткая причина пишутся только в технический журнал.

Сначала verifier принимает окончательное решение по всем кандидатам источника; только затем одобренные факты передаются в Mem0 по одному с `infer=False`. Все Mem0 writes одного источника публикуются в SQLite одной транзакцией. Частичный сбой немедленно пытается удалить новые vectors и восстановить прежние версии. Если компенсацию оборвал новый сбой или остановка процесса, следующий claim сначала находит все Mem0 records по metadata `source_id` и удаляет orphan либо восстанавливает точный текст из SQLite; до успешной reconciliation extraction и новые writes запрещены. Mem0 служит только векторным индексом буквального одобренного текста, включая пробелы. Комбинация `(scope, subject_id, fact_key)` имеет одну актуальную версию; новое ясное самоутверждение обновляет прежний vector ID in-place. Поиск дополнительно сверяет ID и буквальное равенство текста результата с SQLite-реестром одобренных фактов той же области чата.

До завершения соответствующего буферизованного хода источник имеет статус `waiting`, блокирует более новые элементы FIFO и недоступен worker. Только успешное завершение хода после доставки обычного ответа переводит фиксированный снимок ID в `pending`; исключение или отмена оставляет их `waiting`. Поэтому память не задерживает Telegram reply и отменённый debounce-task не освобождает более новые сообщения. TikTok-only сообщение не создаёт LLM-ход чата и освобождает только собственный source ID для пустого окончательного решения extractor'а. Если процесс перезапущен до доставки, оставшийся `waiting`-источник становится `abandoned`, не проверяется и теряет сырой текст. Временная ошибка блокирует более новые источники, чтобы запоздалый retry не перезаписал более свежий факт. После ровно пяти технических сбоев источник становится dead-letter. Полный сырой текст очищается после окончательного решения; для одобренного факта остаются только точная цитата и provenance.

## Последствия

Точность и воспроизводимость важнее полноты: неоднозначные факты будут потеряны, а новые воспоминания появятся с задержкой и потребуют двух LLM-вызовов. Зато сбой extractor'а, verifier'а, Mem0 или Qdrant не создаёт память, чатовые области не смешиваются, а Mem0 не может самостоятельно расширить одобренную формулировку.
